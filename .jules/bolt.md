## 2024-03-24 - Convert Array Includes to Set Has in List Rendering
**Learning:** Found an O(N^2) bottleneck when rendering `ChannelShelf` because `Array.includes()` was being used inside a list mapping over channels for both favorites and recents. Even if N is only ~10,000, calling `Array.includes` on a large array inside another loop creates an O(N * M) rendering overhead that blocks the main thread.
**Action:** Always use `Set.has()` (O(1)) instead of `Array.includes()` when filtering or checking membership inside large map/filter loops in React components. Also added `useMemo` for filtering list items in `App.tsx` (`visibleChannels`) to avoid allocating and re-filtering arrays on every render (especially when renders are triggered by frequent changes like player volume updates).
## 2024-05-24 - V8 Object Allocation & Deoptimization inside Loops
**Learning:** In Javascript, repeatedly cloning an object using the spread operator (`{ ...obj }`) and calling `delete` inside an inner loop causes excessive memory allocations, heavy garbage collection pressure, and frequent deoptimization of hidden classes in the V8 engine, resulting in poor performance.
**Action:** Always verify if an object actually requires modification before copying it and avoid `delete` on non-existent keys. Checking for changes using `hasOwnProperty` and preserving the original object reference if no changes occur avoids unnecessary allocations and maintains engine optimizations.
## 2025-03-09 - Memoized dictionaries vs Lazy lookup hooks
**Learning:** Constructing a massive mapping dictionary inside `useMemo` is expensive when iterating over a very large array (e.g. 50,000 items), blocking the main thread. Switching to a `useCallback` that performs a lazy lookup when a component actually needs the data for rendering drastically reduces up-front CPU cost and memory allocations.
**Action:** When deriving state for large lists where only a small subset is rendered or accessed at any given time, prefer lazy evaluation via parameterized getters (`useCallback`) instead of full pre-computation (`useMemo`).
## 2025-03-11 - Hoisting Loop-Invariant Computations
**Learning:** Performing expensive string normalizations and Set allocations for a constant search query inside an array mapping loop over potentially thousands of items (e.g. searching EPG channels) creates significant, redundant processing overhead that blocks the thread.
**Action:** Always hoist loop-invariant data processing out of mapping/filtering loops to ensure the computation happens exactly once, and pass the pre-computed value down.
